# 사용자 테스트 시나리오

워크스페이스 템플릿 시스템 도입 후 전체 기능 검증용 시나리오.

---

## 사전 조건

- PostgreSQL 실행 중 (graph_viewer DB)
- `alembic upgrade head` 완료
- 백엔드/프론트엔드 서버 기동
- 시드 데이터 투입: `cd backend && .venv/bin/python scripts/seed_test_data.py`

---

## TC-01: 회원가입 및 로그인

**목적**: 인증 플로우 정상 동작 확인

| 단계 | 동작 | 기대 결과 |
|------|------|-----------|
| 1 | `/register` 페이지 접속 | 회원가입 폼 표시 |
| 2 | 이름, 이메일, 비밀번호 입력 후 가입 | 성공 → 로그인 페이지로 이동 |
| 3 | 동일 이메일로 로그인 | 성공 → `/` (그래프 페이지)로 이동 |
| 4 | 비밀번호 틀리게 입력 | 오류 메시지 표시, 로그인 실패 |
| 5 | 토큰 없이 `/query` 접속 | `/login`으로 리다이렉트 |

## TC-02: 기본 템플릿 시드 확인

**목적**: 서버 기동 시 기본 템플릿 자동 생성 확인

| 단계 | 동작 | 기대 결과 |
|------|------|-----------|
| 1 | 로그인 후 `/settings` 이동 | 워크스페이스 설정 페이지 표시 |
| 2 | 템플릿 목록 확인 | "조직관리", "인프라" 2개 템플릿 존재 |
| 3 | "조직관리" 클릭 | 4개 레벨 표시: user(L0) → group(L1) → task(L2) → subtask(L3) |
| 4 | "인프라" 클릭 | 6개 레벨 표시: user(L0) → group(L1) → cluster(L2) → namespace(L3) → workload(L4) → pod(L5) |
| 5 | L0(user), L1(group)이 "고정" 표시 | 수정/삭제 버튼 비활성화 |

## TC-03: 템플릿 생성

**목적**: 사용자가 커스텀 템플릿을 생성할 수 있는지 확인

| 단계 | 동작 | 기대 결과 |
|------|------|-----------|
| 1 | `/settings` → "새 템플릿" 클릭 | 빈 폼 (L0 user, L1 group만 있는 상태) |
| 2 | 이름: "프로젝트 관리", 설명: "프로젝트 추적용" 입력 | 입력 반영 |
| 3 | "+ 레벨 추가" → node_type: `project`, label: `프로젝트`, color 선택 | L2 추가됨 |
| 4 | "+ 레벨 추가" → node_type: `milestone`, label: `마일스톤` | L3 추가됨 |
| 5 | "+ 레벨 추가" → node_type: `issue`, label: `이슈` | L4 추가됨 |
| 6 | "+ 규칙 추가" → source: `group`, target: `project` | 엣지 규칙 1개 |
| 7 | "+ 규칙 추가" → source: `project`, target: `milestone` | 엣지 규칙 2개 |
| 8 | "+ 규칙 추가" → source: `milestone`, target: `issue` | 엣지 규칙 3개 |
| 9 | "저장" 클릭 | 성공 → 템플릿 목록에 "프로젝트 관리" 추가됨 |

## TC-04: 템플릿 수정

**목적**: 기존 템플릿의 커스텀 레벨 수정/삭제 확인

| 단계 | 동작 | 기대 결과 |
|------|------|-----------|
| 1 | TC-03에서 만든 "프로젝트 관리" 선택 | 5개 레벨(L0~L4), 3개 엣지 규칙 표시 |
| 2 | L4(issue)의 label을 "버그"로 변경 | 입력 반영 |
| 3 | "저장" 클릭 | 성공 → L4 label이 "버그"로 변경됨 |
| 4 | L4(issue) 삭제 (X 클릭) | L4 제거, 관련 엣지 규칙(milestone→issue) 자동 제거 |
| 5 | "저장" 클릭 | 성공 → 4개 레벨, 2개 엣지 규칙 |
| 6 | L0(user) 수정 시도 | 입력 필드 비활성화 — 수정 불가 |
| 7 | L1(group) 삭제 시도 | 삭제 버튼 없음 — 삭제 불가 |

## TC-06: 템플릿 검증 (백엔드)

**목적**: Pydantic 검증 규칙 동작 확인 (API 직접 호출)

| 단계 | 동작 | 기대 결과 |
|------|------|-----------|
| 1 | POST `/api/templates` — levels에 L0 없음 | 400 Bad Request (level 0 = user 필수) |
| 2 | POST — L0이 `user`가 아닌 다른 node_type | 400 (level 0 must be 'user') |
| 3 | POST — L1이 `group`이 아닌 다른 node_type | 400 (level 1 must be 'group') |
| 4 | POST — level 번호 불연속 (0,1,3) | 400 (levels must be consecutive) |
| 5 | POST — 중복 node_type | 400 (duplicate node_type) |
| 6 | POST — edge_rules에 levels에 없는 타입 참조 | 400 (invalid source/target type) |
| 7 | POST — color 형식 오류 (`red` → `#ff0000` 필요) | 400 (color format invalid) |

## TC-08: 노드 CRUD

**목적**: 노드 생성/수정/삭제가 템플릿 기반으로 동작

| 단계 | 동작 | 기대 결과 |
|------|------|-----------|
| 1 | 그래프 페이지 → "+ 노드" 클릭 | NodeForm 모달 표시 |
| 2 | 타입 셀렉트박스 확인 | 활성 템플릿의 levels에 정의된 node_type만 선택 가능 |
| 3 | 타입: `task`, 이름: `새 작업`, 속성 입력 → 저장 | 노드 생성 → 그래프에 표시 |
| 4 | 생성된 노드 클릭 → "수정" | 수정 모달 → 이름/속성 변경 가능 |
| 5 | 노드 "삭제" 클릭 → 확인 | 삭제 → 연결된 엣지도 함께 삭제 |

## TC-10: 벡터 검색

**목적**: 자연어 기반 벡터 유사도 검색 + 1-hop 확장

| 단계 | 동작 | 기대 결과 |
|------|------|-----------|
| 1 | `/query` → "재귀 탐색" 탭 | 검색 입력란 표시 |
| 2 | `프론트엔드 작업` 검색 | 관련 task 노드가 score 순으로 정렬 |
| 3 | 결과에서 depth=0 확인 | 직접 매칭 (벡터 유사도 시드) |
| 4 | 결과에서 depth=1 확인 | 1-hop 엣지 확장으로 도달한 노드 |
| 5 | `JWT 인증` 검색 | 관련 기술/작업 노드 반환 |
| 6 | `kubernetes 배포` 검색 (다른 그래프 데이터) | K8s 관련 노드 반환 |
| 7 | 응답 시간 표시 확인 | 응답 시간(ms) 표시됨 |
